# LAST BID Milestone 2.5 — UX Prototype

현재 Milestone 1과 Milestone 2는 완료된 상태다.

이미 구현된 기능:

- Godot 4.x + GDScript
- 플레이어 1명 + NPC 3명
- 10라운드
- 오름 입찰 경매
- 카드 6장
- 체력, 골드, 카드 보유
- 즉시 효과와 지연 효과
- 심판 단계
- 승패 판정
- 고정 시드
- 디버그 로그
- 카드 정보 비대칭
- 실제 카드 정보와 공개 단서 분리
- 플레이어/NPC별 KnowledgeState
- 정보 토큰과 추가 조사
- 수집가, 채권자, 도박사의 고유 평가 방식
- NPC 대사
- 도박사의 제한적인 허세 입찰
- 같은 시드에서 정보 배분과 AI 행동 재현

이번 작업의 목적은 최종 UI 디자인이 아니라, 현재 구현된 기능이 실제 플레이 화면에서 자연스럽게 읽히는지 검증하기 위한 UX 와이어프레임을 만드는 것이다.

## 핵심 목표

- 각 게임 단계의 화면 구조를 분리
- 플레이어가 현재 무엇을 해야 하는지 명확히 표시
- 카드 정보와 NPC 반응의 우선순위 정리
- PRE_INFO, AUCTION, POST_AUCTION, JUDGMENT 흐름 시각화
- 디버그 로그를 일반 UI에서 분리
- 향후 Milestone 3 기능이 들어갈 자리를 미리 확보
- 기존 게임 로직과 결정론적 RNG는 변경하지 않음

## 제외 사항

이번 작업에서는 다음을 하지 않는다.

- 최종 픽셀 아트
- 완성형 일러스트
- 복잡한 애니메이션
- 사운드
- Steam 대응
- 협상
- 거래
- 평판
- 카드 이전
- 플레이어 직업
- 상점
- 메타 진행
- 새 카드 추가
- 게임 밸런스 대규모 조정
- 기존 AI 로직 리팩터링

이번 목표는 어디까지나 **플레이 가능한 와이어프레임 UI**다.

## 1. 전체 UI 방향

기본 시각 스타일:

- 배경: 거의 검정에 가까운 짙은 색
- 패널: 짙은 회색
- 주요 강조: 낡은 금색
- 위험/피해: 어두운 적색
- 기본 텍스트: 아이보리
- 보조 텍스트: 낮은 대비의 회색

색상은 임시값이어도 되지만, 코드 전체에 직접 흩뿌리지 말고 Theme 또는 공통 스타일 리소스로 관리해줘.

최종 디자인처럼 보이게 만들 필요는 없지만, 단계별 화면 구분과 정보 위계는 명확해야 한다.

## 2. 레이아웃 원칙

전체 UI는 Control/Container 기반으로 구성해줘.

- Anchor와 Container를 사용
- 절대 좌표 남용 금지
- 16:9 데스크톱 기준
- 최소 1280x720에서도 깨지지 않게
- 1920x1080에서 자연스럽게 확장
- 텍스트가 길어져도 레이아웃이 무너지지 않게
- 모든 주요 패널은 최소 크기와 확장 규칙을 가짐
- 임시 UI라도 한 화면에 모든 정보를 억지로 넣지 않음

권장 구조:

- 상단: 현재 라운드, 현재 단계, 시드, 남은 정보 토큰
- 좌측: 참가자 상태
- 중앙: 현재 카드와 핵심 정보
- 우측: NPC 반응 또는 단계별 보조 패널
- 하단: 주요 액션 버튼

현재 프로젝트 구조를 먼저 보고, 기존 씬을 최대한 재사용해줘.

## 3. 공통 상단 HUD

모든 주요 단계에서 공통으로 표시:

- 현재 라운드 / 총 라운드
- 현재 단계
- 현재 시드
- 남은 정보 토큰
- 디버그 패널 토글 버튼

예:

```text
ROUND 3 / 10
PRE_INFO
SEED 102938
INFO 1
```

일반 플레이 UI에서는 내부 enum 이름을 그대로 노출하지 말고 사용자용 텍스트로 변환해줘.

예:

- PRE_INFO → 사전 정보
- AUCTION → 경매
- POST_AUCTION → 낙찰 후 처리
- JUDGMENT → 심판
- ROUND_END → 라운드 종료

## 4. 참가자 패널

좌측에 플레이어와 NPC 3명의 상태를 보여줘.

각 참가자 카드에 표시:

- 이름
- 아키타입
- 체력
- 골드
- 생존 여부
- 현재 경매 상태
  - 입찰 가능
  - 최고 입찰자
  - 패스
  - 사망
- 최근 대사 1줄
- 현재 차례 강조

플레이어는 NPC와 시각적으로 구분해줘.

예:

- 플레이어: 테두리 강조
- 최고 입찰자: 금색 강조
- 패스: 낮은 불투명도
- 사망: 붉은 회색 처리
- 현재 차례: 밝은 외곽선 또는 작은 화살표

NPC 내부 평가 점수나 허세 여부는 일반 UI에 표시하지 않는다.

## 5. PRE_INFO 화면

PRE_INFO는 카드의 정보를 확인하고 추가 조사 여부를 결정하는 단계다.

중앙 카드 패널에 표시:

- public_name
- 카드 이미지 자리
- 공개된 역할군
- 위험도 범위
- 가치 범위
- 발동 시점
- 대상 유형
- 현재까지 공개된 단서 목록
- 조사로 새로 얻은 단서가 있다면 별도 강조

우측 반응 패널:

- 수집가의 첫 반응
- 채권자의 첫 반응
- 도박사의 첫 반응

대사는 각 NPC당 1줄만 보이게 하고, 너무 긴 경우 자동 줄바꿈 또는 말줄임 처리.

하단 버튼:

- 추가 조사
- 경매 시작

추가 조사 버튼:

- 남은 정보 토큰 표시
- 토큰이 0이면 비활성
- 모든 단서를 이미 아는 경우 비활성
- 버튼 클릭 시 새 단서를 짧게 강조
- 조사 결과가 로그에만 묻히지 않게 화면에 직접 표시

PRE_INFO에서는 입찰 버튼을 표시하지 않는다.

## 6. AUCTION 화면

AUCTION은 실제 심리전의 핵심 화면이다.

중앙 카드 영역:

- public_name
- 카드 이미지 자리
- 현재까지 공개된 단서
- 정확한 효과는 숨김
- 조사로 얻은 단서도 유지

경매 정보 영역:

- 시작가
- 현재가
- 다음 최소 입찰가
- 최고 입찰자
- 현재 차례
- 최소 인상폭

현재가가 없을 때는 `0 G` 대신 `입찰 없음`으로 표시.

첫 입찰 버튼은:

```text
첫 입찰 200 G
```

이후에는:

```text
250 G로 입찰
```

처럼 표시.

플레이어 액션:

- 입찰
- 패스

규칙:

- 플레이어 차례가 아니면 비활성
- 패스한 뒤에는 다시 활성화되지 않음
- 보유 골드보다 높은 입찰은 비활성
- 행동 불가 이유를 작은 보조 문구로 표시

NPC 반응:

- 각 NPC 참가자 카드에 최근 대사 1줄
- 입찰 시 가격 변화가 눈에 보이게 갱신
- 패스한 NPC는 상태를 명확히 표시
- 현재 차례를 강조

복잡한 애니메이션은 금지한다.

간단한 Tween 정도로 숫자 변화와 패널 강조만 허용한다.

## 7. POST_AUCTION 화면

현재 Milestone 3 기능은 아직 없지만, 낙찰 결과를 보여주는 UX 뼈대를 만든다.

표시:

- 낙찰된 카드 public_name
- 낙찰자
- 낙찰 가격
- 카드 공개 상태
- 현재 소유자
- 다음 단계 안내

미래 기능 자리:

- 개봉
- 보관
- 판매
- 소각

이 버튼들은 이번 Milestone에서는 비활성 상태로 표시하거나 `준비 중` 배지를 붙여도 된다.

중요:

- 비활성 버튼이 실제 기능처럼 오해되지 않게
- 현재는 `계속` 또는 `심판으로` 버튼만 실제 작동
- 향후 기능을 붙이기 쉬운 구조로 분리

## 8. JUDGMENT 화면

JUDGMENT에서는 로그를 읽지 않아도 결과를 이해할 수 있어야 한다.

표시:

- 발동한 카드 목록
- 실제 공개된 카드명
- 발동 대상
- 체력 변화
- 골드 변화
- 사망 여부
- 소비된 카드 여부
- 심판 결과 요약

예:

```text
검은 장부 발동

채권자 +80 G
도박사 -1 HP
```

여러 효과가 있으면 순서대로 카드 형태 또는 결과 리스트로 보여줘.

하단 버튼:

- 다음 라운드
- 게임 종료 시 결과 화면으로

이번에는 복잡한 연출 없이 텍스트와 상태 변화만 명확히 보여줘.

## 9. ROUND_END / RUN_RESULT

### ROUND_END

- 이번 라운드 요약
- 현재 생존자
- 다음 라운드 안내
- 자동 진행 또는 계속 버튼

### RUN_RESULT

- 승리 / 패배
- 종료 라운드
- 최종 HP
- 최종 골드
- 획득 카드
- 사용한 정보 토큰
- 사용 시드
- 다시 시작
- 같은 시드로 다시 시작
- 새 시드로 시작

## 10. 디버그 패널 분리

현재 디버그 로그가 일반 화면을 과도하게 차지하지 않도록 별도 패널로 분리해줘.

형태:

- 우측 슬라이드 패널
- 팝업
- 접이식 하단 패널

중 하나로 구현.

디버그 패널 표시 내용:

- 실제 카드명
- 실제 effects
- 모든 단서
- 참가자별 KnowledgeState
- NPC 주관적 가치
- 최대 입찰가
- 허세 여부
- RNG 관련 정보
- 디버그 로그

일반 플레이 기본 상태에서는 닫혀 있어야 한다.

## 11. 화면 전환 구조

기존 GameFlowController가 단계 전환의 유일한 권한을 유지해야 한다.

UI는 phase를 직접 바꾸지 않는다.

권장 구조:

- MainGameUI
- TopHUD
- ParticipantPanel
- CardInfoPanel
- AuctionPanel
- ReactionPanel
- JudgmentPanel
- DebugPanel
- ResultPanel

각 패널은 현재 phase에 따라 표시/숨김 처리한다.

가능하면 한 씬 안에서 패널을 전부 뒤엉키게 만들지 말고, 재사용 가능한 서브씬으로 분리해줘.

## 12. 사용자 피드백

버튼 클릭과 상태 변화에 최소한의 피드백을 넣어줘.

- 버튼 hover
- disabled 상태
- 현재 차례 강조
- 새 단서 공개 강조
- 입찰가 갱신
- 최고 입찰자 변경
- 패스 상태 반영
- 체력/골드 변화 강조
- 사망 상태 반영

복잡한 연출보다 명확성이 우선이다.

## 13. 테스트

기존 Milestone 1, 2 테스트를 유지해줘.

추가로 가능한 범위에서 다음을 검증해줘.

- PRE_INFO에서 입찰 버튼이 보이지 않음
- AUCTION에서 조사 버튼이 보이지 않음
- FULLY_REVEALED 전에는 실제 효과가 일반 UI에 노출되지 않음
- 디버그 패널을 열었을 때만 실제 효과 표시
- 플레이어 차례가 아닐 때 입찰 버튼 비활성
- 패스 후 입찰 버튼 비활성
- 골드 부족 시 입찰 버튼 비활성
- 정보 토큰 0일 때 조사 버튼 비활성
- 조사 후 새 단서가 UI에 반영
- 최고 입찰자 상태가 참가자 패널에 반영
- JUDGMENT 결과가 로그 없이도 UI에 표시
- phase 변경 시 올바른 패널만 표시
- 동일 시드로 다시 시작 기능 정상 작동

UI 테스트 자동화가 어려운 부분은 수동 검증 절차를 README에 작성해줘.

## 14. 완료 조건

Milestone 2.5 완료 조건:

- PRE_INFO, AUCTION, POST_AUCTION, JUDGMENT, RUN_RESULT 화면이 구분됨
- 각 단계에서 필요한 정보와 버튼만 표시됨
- 플레이어가 현재 해야 할 행동을 이해할 수 있음
- NPC 상태와 최근 반응을 한눈에 확인 가능
- 카드 단서와 조사 결과가 로그 없이도 읽힘
- 경매 상태와 현재 차례가 명확함
- 심판 결과를 로그 없이도 이해 가능
- 디버그 정보가 일반 UI와 분리됨
- 기존 Milestone 1, 2 기능이 깨지지 않음
- 1280x720과 1920x1080에서 레이아웃이 무너지지 않음
- 최종 아트 없이도 전체 플레이 흐름을 검증할 수 있음
