## LAST BID

**LAST BID**는 위험한 물건들이 거래되는 비밀 경매장을 배경으로 한 싱글 플레이 심리 경매 게임입니다.

플레이어는 경매에 등장한 카드의 정확한 효과를 처음부터 알 수 없습니다.  
제한적으로 공개된 단서와 상대 참가자들의 반응, 입찰 패턴, 성향을 관찰해 카드의 가치와 위험을 추론해야 합니다.

높은 가격에 낙찰받는 것이 항상 좋은 선택은 아닙니다.  
어떤 물건은 당장의 이익을 주지만 이후 더 큰 대가를 요구하고, 어떤 물건은 다른 참가자의 자산이나 생존 상태에 따라 전혀 다른 결과를 만듭니다.

플레이어는 매 라운드 다음을 판단하게 됩니다.

- 이 물건은 어떤 효과를 가진 카드인가
- 상대는 왜 이 카드에 관심을 보이는가
- 지금 가격까지 따라갈 가치가 있는가
- 이번 입찰을 포기하는 것이 더 나은가
- 현재의 이익과 미래의 위험 중 무엇을 선택할 것인가

## 핵심 특징

### 불완전한 카드 정보

카드는 처음부터 실제 이름과 정확한 효과를 전부 공개하지 않습니다.

플레이어는 역할군, 위험도, 가치 범위, 발동 시점, 대상 유형과 같은 제한된 단서를 통해 카드의 정체를 추론합니다.

추가 조사를 사용하면 더 많은 정보를 얻을 수 있지만, 사용할 수 있는 정보 자원은 제한되어 있습니다.

### 서로 다른 AI 참가자

플레이어는 각기 다른 성향을 가진 세 명의 AI 참가자와 경쟁합니다.

- **마라 — 겁 많은 생존주의자**
  위험을 피하고 생존에 도움이 되는 카드와 안전한 거래를 선호합니다.

- **세라 — 정보 브로커**
  정보와 계약의 가치를 계산하며 자신만 아는 단서를 거래에 활용합니다.

- **볼트 — 공격적 투기꾼**
  고위험·고수익 카드를 선호하며 도발, 허세 입찰, 공격적인 협상을 시도합니다.

각 참가자는 서로 다른 정보를 가지고 있으며, 자신이 알고 있는 단서와 성향에 따라 카드의 가치를 다르게 평가합니다.

### 위험한 경매

경매에 등장하는 카드들은 단순한 보상 아이템이 아닙니다.

일부 카드는 골드를 제공하지만 일정 시간이 지나면 피해를 주고, 일부 카드는 다음 심판 단계나 다음 경매의 규칙을 바꿉니다.

카드의 가치는 현재 체력, 골드, 보유 카드, 남은 라운드와 상대 참가자의 상태에 따라 계속 달라집니다.

## 실행 방법

Godot 4.7 stable에서 프로젝트를 열거나 다음 명령으로 실행합니다.

```bash
godot --path .
```

macOS에서 `godot`이 PATH에 없다면:

```bash
/Applications/Godot.app/Contents/MacOS/Godot --path .
```

메인 씬은 `scenes/main.tscn`이며 기본 해상도는 1280×720입니다.

## 현재 개발 상태

현재 안정 기준선은 **Milestone 6 비주얼 프로토타입·온보딩·UI 정리 완료** 상태입니다.

- 경매, 정보 비대칭, 낙찰 후 위험 관리, 캐릭터 협상과 약속 시스템이 하나의 10라운드 런으로 연결되어 있습니다.
- 공통 Theme과 3열 Game Shell, 카드·봉인·참가자 컴포넌트, 단계별 화면 위계로 텍스트 와이어프레임을 플레이 가능한 비주얼 프로토타입으로 전환했습니다.
- 메인 메뉴, 첫 런 Coach Mark, 용어 Tooltip, 약속 위반 확인, Toast와 짧은 피드백 연출을 제공합니다.
- 설정에서 창/전체 화면, 해상도, UI 배율, 텍스트 크기, Reduce Motion, 튜토리얼 재생과 DEBUG 표시 여부를 선택할 수 있습니다.
- 캐릭터는 교체 가능한 TextureRect와 단색 실루엣을 사용합니다. 최종 초상화·카드 일러스트·폰트·오디오는 아직 포함하지 않습니다.

Milestone 1~6에서 구현한 범위와 단계별 설계 결정은 [마일스톤 기록](./docs/MILESTONES.md)에서 관리합니다.

### 검증

자동 검증:

```bash
godot --headless --path . -s res://tests/test_runner.gd
```

Milestone 1~5 회귀 테스트와 함께 Theme·공통 컴포넌트, 메뉴·설정·온보딩, 필수 용어 Tooltip, 봉인 표현, 약속 위반 경고, 4개 해상도·3개 UI 배율, 활성 약속 3개 레이아웃과 기존 20 Seed 자동 런을 포함한 `838 assertions`를 검증합니다.

### 지속적 통합

GitHub Actions의 `Godot 4.7 regression` 체크는 `main` 대상 PR, `main` push, 수동 실행에서 동작합니다.

- 공식 Godot `4.7-stable` Linux 바이너리를 고정 SHA-256으로 검증한 뒤 사용
- 변경 범위의 공백 오류 검사
- 에디터 헤드리스 import로 씬과 리소스 로드 확인
- 회귀 테스트를 서로 독립된 프로세스에서 2회 실행해 결정론 재확인
- 같은 PR의 이전 실행은 취소해 불필요한 Actions 사용 시간 절약
- 저장소 쓰기나 비밀값 없이 `contents: read` 권한만 사용

워크플로 정의는 `.github/workflows/ci.yml`에 있습니다. `main` 보호 규칙에서는 `Godot 4.7 regression`을 필수 체크로 지정하는 것을 권장합니다.

수동 UX 검증 절차:

1. 프로젝트를 실행해 메인 메뉴의 새 게임, 같은 Seed, Seed 입력, 설정과 종료가 표시되는지 확인합니다.
2. 첫 새 게임에서 PRE_INFO Coach Mark가 한 번에 하나만 표시되고 건너뛰기·다시 보지 않기가 동작하는지 확인합니다.
3. 1280×720에서 참가자 4명, 카드, 상황 패널과 하단 액션이 한 화면에 들어오고 `+`로 Relationship·Reputation·Tell·기억을 펼칠 수 있는지 확인합니다.
4. `추가 조사` 후 금색 `◆ 새 조사 단서`와 INFO 감소를 확인하고 위험도·봉인·Tell 등 주요 용어의 Tooltip을 확인합니다.
5. 협상에서 NPC 이름·감정·Tell과 보상·요구·기한·위반 조건이 분리되어 읽히는지 확인합니다.
6. 모든 제안을 처리하기 전에는 경매 시작이 비활성이고, 재제안이 제안당 한 번만 가능한지 확인합니다.
7. 경매에서 `입찰 없음`, `첫 입찰 N G`, 현재 차례, 최고 입찰자와 입찰 불가 안내가 즉시 갱신되는지 확인합니다.
8. 입찰 금지 약속을 수락한 뒤 입찰을 눌러 위반 경고를 확인하고, 취소와 강행이 모두 가능한지 확인합니다.
9. 직접 낙찰한 뒤 봉인 세 슬롯, 다음 사고 확률, 보관·판매·소각을 확인하고 처리 전에는 `심판으로`가 비활성인지 확인합니다.
10. 세 번째 봉인까지 열어 실제 카드명·효과, 카드 공개 강조와 `ON_OPEN` 결과가 표시되는지 확인합니다.
11. 심판 결과가 카드 효과 → 약속 결과 → 사망 순서로 표시되고 RUN_RESULT에 카드·약속·평판·배신·Seed 요약이 나타나는지 확인합니다.
12. 설정에서 1280×720, 1600×900, 1920×1080, 2560×1440와 UI 80·100·120%, 텍스트 크기를 바꿔 잘림과 버튼 이탈이 없는지 확인합니다.
13. Reduce Motion을 켜고 단계 전환·가격·피해 피드백이 입력을 막지 않으며 상태 텍스트와 Toast는 유지되는지 확인합니다.
14. DEBUG를 켜 실제 카드 정보, 숨겨진 목표, 제안·배신 점수와 RNG가 일반 UI와 분리되는지 확인합니다.
15. 동일 Seed로 재시작해 카드 순서, 약속 제안, NPC 배신 여부와 결과 순서가 재현되는지 확인합니다.

## 개발 및 Git 워크플로

저장소는 `main` 기반의 짧은 작업 브랜치와 Conventional Commits, Squash and merge를 사용합니다.

- 브랜치: `feat/post-auction-actions`, `fix/auction-turn-lock`
- 커밋·PR: `feat(ui): add phase-specific auction panels`
- 자동화 브랜치: `codex/<type>-<short-description>`

전체 브랜치·커밋·PR·릴리스 규칙은 [CONTRIBUTING.md](./CONTRIBUTING.md)를 따릅니다. PR 작성 시 저장소 템플릿의 테스트 및 결정론적 RNG 체크리스트를 완료해야 합니다.
